21 Questions · Three Tiers
Tier I Orientation
01What is ImmunoVerse-Chat, in one sentence?

A conversational agent that gives researchers natural-language access to the ImmunoVerse pan-cancer immunopeptidome atlas — 28,446 antigens across 21 tumor types and 11 molecular event classes — through 55+ schema-validated tools, with two layers of anti-hallucination enforcement so every number in the response is traceable back to the atlas.

02What problem does it actually solve?

The ImmunoVerse atlas is rich but hard to use. Querying it required writing code against multi-GB spreadsheets, understanding the scoring model, and knowing which of several sub-datasets to pull from. Clinicians and cancer biologists don't do that. ImmunoVerse-Chat collapses the distance between the question a biologist actually asks ("what are tumor-specific targets in neuroblastoma that aren't in critical normal tissues?") and the ranked peptide list the atlas can produce.

03What data does it sit on?

Primary: the ImmunoVerse/NeoVerse pan-cancer immunopeptidome atlas (Li, Guzmán-Bringas, Sharma et al., bioRxiv 2025) — 28,446 antigens, 21 cancer types, MS-validated peptide–HLA evidence, tumor and normal expression, HLA population frequencies, non-canonical-ORF annotations, and gene-essentiality data.

External (live API): IEDB for curated epitope evidence, ClinicalTrials.gov for gene-level trial status, RCSB PDB for crystal and cryo-EM structures.

User data: TesOrAI mass-spec classification outputs — users can upload their own MS results (up to 30 files, TSV/CSV/XLSX, or direct from Dropbox/Google Drive URLs) and cross-reference them against the atlas in-session.

04Who built it? Is it available?

Built by Aman Sharma at NYU under Prof. Mark Yarmarkovich, with Dr. Guangyuan (Frank) Li as computational mentor and co-author on the underlying atlas, and Dr. Michele Palamenghi as a contributor.

Deployed on NYU HPC (SLURM). Not yet publicly available. The atlas is covered by PCT/US2025/019387. Access or licensing inquiries go through NYU Technology Opportunities and Ventures.

05Who is the intended user?

Three groups. Cancer biologists and clinicians who want to interrogate the atlas without writing code. Computational researchers who want reproducible, auditable analytical paths they can cite. Wet-lab immunologists who generate their own mass-spec data in TesOrAI and want to validate their peptides against the pan-cancer atlas before committing to a target.

06What can I actually do with it? Give me three examples.

1. Target discovery. "Find tumor-specific peptides in neuroblastoma that avoid critical normal tissues." The agent ranks candidates by the composite therapeutic score, flags safety tiers, and shows dual HLA views (by U.S. population frequency and by affinity).

2. Evidence check for a known peptide. "Tell me more about QYNPIRTTF." The agent pulls atlas scoring, queries IEDB for published T-cell assays, checks ClinicalTrials.gov for trials targeting the parent gene, and searches RCSB for crystal structures — each layer explicitly labeled as peptide-level or gene-level.

3. Structural analysis. "Import PDB 8EK5 and score the binding." The agent fetches the structure, auto-classifies chains (HLA / peptide / binder / β2m), scores binder–peptide and binder–HLA contacts separately (salt bridges, H-bonds, π–π, cation–π, hydrophobic, disulfide), and renders a 3Dmol.js viewer in the browser.

Tier II How It Compares
08Why not just use ChatGPT on the atlas?

A general LLM given spreadsheet data will confidently invent numbers. Try it: upload a 28,000-row peptide atlas to ChatGPT and ask for the top neuroblastoma targets. You will get plausible-looking peptide sequences, scores, and HLA alleles. A non-trivial fraction of them will not exist in your data.

ImmunoVerse-Chat forbids the LLM from generating biological content. The LLM's role is to parse intent and pick a tool. Every number, peptide sequence, HLA allele, and score in the response comes from a tool call against the atlas — and every number is cross-checked after the LLM writes it. If the LLM hallucinates a number, the system detects it and forces a regeneration before the user sees anything.

09Isn't this just RAG over the atlas?

No. RAG retrieves text chunks and hands them to the LLM, which then writes prose conditioned on those chunks. That is still free-form generation — the LLM can still misread a number, combine chunks incorrectly, or hallucinate a relationship the source does not support.

ImmunoVerse-Chat retrieves structured data from code, not chunks from text. The LLM receives the output of a validated function — a ranked list of peptides with their scores, a scoring breakdown, a safety tier — and its job is only to narrate what the function returned. Additionally, a Python-side interpretation-hints layer pre-computes the scoring explanation in advance, so the LLM does not derive the interpretation itself; it reports it.

A future literature-RAG augmentation for citation is on the roadmap, but it will be a secondary layer for finding supporting papers, not the primary reasoning path.

10Is this built on LangChain or LangGraph?

Neither. The orchestrator is a custom agent loop implemented directly in Python. It uses litellm for model-agnostic LLM calls with a fallback chain (gpt-4o-mini → gpt-4o → gemini-2.0-flash → gpt-4-turbo), a decorator-based tool-registration pattern for the 55+ tools, JSON-Schema validation on every call, and a custom trace logger that writes JSONL events with code hashes for reproducibility.

LangGraph is a fine framework for general agent workflows, but it adds abstraction layers that get in the way when the priority is deterministic, auditable scientific tool calls. Writing the loop directly gives full control over validation gates, the post-response number checker, and the replay-from-trace guarantee.

The system does expose an MCP (Model Context Protocol) server alongside its other interfaces, so external LangGraph or Claude-side clients can consume the tool registry through a standard protocol if they want to.

11How does it compare to a vector-database-plus-LLM stack like ChromaDB + GPT-4?

Different problem. Vector DBs are designed for semantic retrieval over unstructured text. Peptide atlases are structured tabular data with numeric fields (tumor specificity, abundance, PSM count, HLA binding affinity) that require deterministic filtering and ranking — not nearest-neighbor embedding search.

If a user asks "find highly tumor-specific peptides with no expression in critical normal tissues," a vector DB cannot answer that correctly. A SQL-style filter over the structured atlas can. ImmunoVerse-Chat chooses the second path.

12What is the advantage over writing a traditional query interface (SQL, REST)?

Two things. First, the biology is encoded in tool chaining, not in the query. A user asking "which targets are safest for an HLA-A*02:01 patient population with no trials yet?" requires filtering the atlas, joining with HLA frequency, excluding genes with existing Phase I+ trials from ClinicalTrials.gov, and ranking by the composite score. That's five different data sources and a non-trivial biological reasoning chain. Expressing it in SQL requires the user to already know every join. The agent resolves it through multi-tool chaining.

Second, context accumulation. The user can say "now show me the best binders by affinity for the top three" and the agent remembers the prior result set. REST endpoints are stateless.

Tier III Methodology & Limits
13How exactly do you prevent hallucination? What's the actual mechanism?

Two layers, both enforced in code.

Layer 1 — Hybrid pre-computation. When a peptide query returns, the system pre-computes all interpretive numbers in Python: fold-change ratios, sample coverage percentages, abundance comparisons between candidates, per-peptide safety context. These are passed to the LLM as structured facts. The LLM writes prose around them but cannot derive them itself.

Layer 2 — Post-response number validation. After the LLM generates a response, a validator extracts every number from the text and cross-checks it against the set of numbers that came out of the tool call. If any number in the response doesn't match the tool output, the response is suppressed, a correction request goes back to the LLM, and it regenerates. The user never sees the unvalidated first draft.

14What is in the tool registry? What do the 55+ tools actually do?

They fall into six categories: atlas query tools (ranking, filtering, gene lookup, tumor-specificity checks, plus live integration with IEDB and ClinicalTrials.gov); analysis tools for user-uploaded MS data, with load / batch / merge / filter / summary / QC operations; cross-referencing tools that compare user-uploaded peptides against the atlas; HLA typing and SLURM cluster-control tools for running OptiType pipelines on sequencing data; PDB binding-analysis tools via RCSB plus a structural interaction scorer; and export tools producing CSV, XLSX, JSON, or Markdown reports.

Each tool is registered through a decorator, carries a JSON schema for its arguments, and is dispatched through a single validated entry point.

15How do you rank candidates?

Peptides are ranked by a composite therapeutic score that integrates multiple lines of evidence: how tumor-specific the peptide is, how broadly it is presented across patient samples, how abundantly it appears on the cell surface, the quality of the mass-spec evidence behind it, and a tiered safety signal based on presentation in normal tissues. Bonuses reward cross-cancer breadth and prior clinical validation; penalties scale with the sensitivity of the normal tissues involved.

The exact weights, components, and the tumor-specificity derivation are part of work that has not yet been published or disclosed. I'm happy to walk through the reasoning in person under a non-disclosure arrangement.

16How is the agent loop structured?

On each user turn: the LLM is given the system prompt, conversation history, and the tool registry; it emits a tool call; the dispatcher validates arguments against the JSON schema and executes the tool; results and interpretation hints are appended to the conversation; the LLM responds with either another tool call or a final answer. The post-response validator then runs on any final answer that followed a peptide query. If validation fails, the loop re-enters for a correction turn.

Token budgets scale dynamically with result size, which keeps small responses fast and prevents large response truncation.

17How do you handle conflicting or ambiguous queries?

Every tool has a JSON schema for arguments. If the LLM proposes a tool call with missing required fields or out-of-range values, the dispatcher rejects it before execution and the LLM receives the validation error and tries again. For biologically conflicting constraints — for example, asking for tumor-specific peptides but also specifying a housekeeping gene by name — the tool returns the filtered result along with an interpretation hint explaining the conflict, rather than silently resolving it.

18What about non-canonical ORFs and dark-genome targets?

The atlas explicitly includes nuORF (novel upstream ORF) peptides, and the scoring rewards them because they are less likely to have induced central tolerance. Non-canonical source-identity tags are first-class filter criteria in both the atlas query tools and the user-upload analysis.

The system consumes nuORF annotations produced upstream; it does not itself perform de novo ncORF discovery from Ribo-seq. That's a separate discovery pipeline (RiboClone) that would feed new candidates into the atlas.

19How do you benchmark it?

Three axes. Tool selection accuracy: does the LLM pick the right tool for a given natural-language query? Measured against a curated set of gold-standard queries with known correct tool paths. Number fidelity: the post-response validator acts as a continuous regression test — any response that slips a fabricated number past the first draft is caught. Replayability: every tool call is logged to JSONL with the code hash of the tool version, so any historical response can be re-run against the same code state and must reproduce.

I'm not citing specific precision numbers at the poster. If you want the benchmark tables, email me.

20What are the failure modes?

Four, honestly.

1. Atlas coverage. Some cancer types are underrepresented in the MS data. Queries against them return small result sets with low confidence, and the agent reports the sample count explicitly ("3/5 samples") rather than pretending the signal is stronger than it is.

2. MS sensitivity floor. Absence of a peptide in the immunopeptidome is not evidence of absence on tumor cells. A peptide can be presented below the MS detection limit. The agent does not claim negative evidence.

3. Gene-level vs peptide-level conflation. ClinicalTrials.gov returns gene-level trial information. The agent always explicitly labels this — a PHOX2B trial does not confirm the specific peptide QYNPIRTTF is being targeted.

4. External API staleness. IEDB and ClinicalTrials.gov are queried live; results reflect the database state at the query moment, which is timestamped in the trace.
